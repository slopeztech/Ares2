# Static Analysis Guide

This document describes how to run and interpret the two static analysis tools
used in ARES: **Cppcheck** and **clang-tidy**.

Both tools operate on the `src/` tree only and save evidence artefacts to
`evidence/` for traceability.

---

## Prerequisites

| Tool | Version | Location |
|------|---------|----------|
| Cppcheck | ≥ 2.13 | `C:\Program Files\Cppcheck\cppcheck.exe` |
| clang-tidy | ≥ 17 (LLVM 22 installed) | `C:\Program Files\LLVM\bin\clang-tidy.exe` |
| Python 3 | ≥ 3.8 | Required by Cppcheck MISRA addon |

Both tools use `compile_commands.json` (generated by PlatformIO's `compiledb`
target) for accurate include paths and compiler flags.

---

## 1. Cppcheck

Cppcheck is a static analyser focused on C/C++ defects. ARES uses two profiles:

### 1.1 Practical profile (no MISRA)

Runs `warning`, `style`, `performance`, and `portability` checks on `src/`.
Fastest and most actionable — intended for day-to-day development.

**Run from VS Code:**

```
Terminal › Run Task › Cppcheck: Practical (no MISRA addon)
```

Or from the command line:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .vscode/scripts/run_cppcheck_practical.ps1
```

**Output:** `evidence/latest/cppcheck-practical.txt`

### 1.2 MISRA profile

Runs the Cppcheck MISRA C 2012 addon against the full `compile_commands.json`
database. Requires a prior build to produce a fresh database.

**Run from VS Code (full pipeline):**

```
Terminal › Run Task › MISRA: Build + Cppcheck
```

Steps executed in order:

1. `Build: PlatformIO` — firmware build (generates object files)
2. `MISRA: Generate compile database` — produces `compile_commands.json`
3. `MISRA: Cppcheck (MISRA addon)` — runs cppcheck with the MISRA addon

**Output:** `evidence/latest/cppcheck-misra-addon.txt`

**Run the MISRA step alone** (when `compile_commands.json` is already fresh):

```
Terminal › Run Task › MISRA: Cppcheck (MISRA addon)
```

### 1.3 Inline suppressions

To silence a specific finding in source code use:

```cpp
// cppcheck-suppress <check-id>
someFunction();
```

Example:
```cpp
// cppcheck-suppress misra-c2012-12.3
for (int i = 0, j = 0; i < n; i++, j++) { ... }
```

## 2. clang-tidy

clang-tidy is a clang-based linter that catches a broader set of C++ issues
than Cppcheck: uninitialized variables, narrowing conversions, readability
anti-patterns, and more. It uses the `.clang-tidy` config at the project root.

### 2.1 Active check groups

| Group | Purpose |
|-------|---------|
| `bugprone-*` | Common bug patterns (integer overflow, suspicious casts, etc.) |
| `cppcoreguidelines-*` (subset) | Narrowing conversions, uninitialized vars, slicing |
| `misc-const-correctness` | Missing `const` on values and references |
| `modernize-use-nullptr/override/using` | C++11/14/17 idioms |
| `performance-*` | Unnecessary copies, value-parameter misuse |
| `readability-*` | Function size, braces, simplifications |

Checks that generate excessive noise on Arduino/ESP-IDF headers or are
inapplicable to embedded code (magic-number policies, ownership semantics,
Fuchsia rules) are disabled. See `.clang-tidy` at the project root for the
full list.

### 2.2 Run from VS Code

**Analyze only** (requires fresh `compile_commands.json`):

```
Terminal › Run Task › clang-tidy: Analyze
```

**Full pipeline** (build → compile database → analyze):

```
Terminal › Run Task › clang-tidy: Build + Analyze
```

### 2.3 Run from command line

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .vscode/scripts/run_clang_tidy.ps1
```

Or invoke clang-tidy directly on a single file:

```powershell
& "C:\Program Files\LLVM\bin\clang-tidy.exe" `
    -p . `
    --extra-arg='-Wno-unknown-warning-option' `
    src/ams/mission_script_engine.cpp
```

### 2.4 Output

Results are saved to:

```
evidence/<date>/clang-tidy-<HHmmss>.txt   ← timestamped archive
evidence/<date>/clang-tidy.txt            ← canonical for that date
evidence/latest/clang-tidy.txt            ← always the most recent run
```

The script exits with code `1` if any `error:`-level findings are reported,
which marks the VS Code task as failed.

### 2.5 Diagnostic format

clang-tidy produces output in the familiar GCC/clang style:

```
src/ams/mission_script_engine.cpp:412:9: warning: variable 'foo' is not initialized [cppcoreguidelines-init-variables]
```

VS Code parses this format automatically via the `problemMatcher` in the task
definition, so findings appear in the **Problems** panel.

### 2.6 Suppressing a finding inline

Use a `NOLINT` or `NOLINTBEGIN`/`NOLINTEND` comment:

```cpp
int x = getValue();  // NOLINT(cppcoreguidelines-init-variables)

// NOLINTBEGIN(readability-braces-around-statements)
if (cond) doSomething();
// NOLINTEND(readability-braces-around-statements)
```

To suppress all checks on a line:

```cpp
doLegacyThing();  // NOLINT
```

Use suppressions sparingly — prefer fixing the root cause.

## 3. Which tool to use when

| Scenario | Recommended tool |
|----------|-----------------|
| Quick check before committing | `Cppcheck: Practical` |
| Pre-release MISRA audit | `MISRA: Build + Cppcheck` |
| C++ idiom and code-quality review | `clang-tidy: Analyze` |
| Full traceability evidence run | All three pipelines |

## 4. Evidence artefacts

All runs are preserved in `evidence/` with date-stamped subdirectories:

```
evidence/
├── latest/                   ← symlink-equivalent: most recent run of each tool
│   ├── cppcheck-misra-addon.txt
│   ├── cppcheck-practical.txt
│   └── clang-tidy.txt
└── 2026-04-29/
    ├── cppcheck-misra-addon-143012.txt
    ├── cppcheck-practical-143120.txt
    └── clang-tidy-143200.txt
```

These files form the static analysis evidence package for DO-178C / process
traceability purposes.

## 5. Modifying the check configuration

### Cppcheck suppressions (project-wide)

Add a line to the `$cppArgs` array in the relevant script:

```powershell
'--suppress=<check-id>:<path>'
```

### clang-tidy checks

Edit `.clang-tidy` at the project root. The `Checks` key uses a comma-separated
list where `-` prefix disables a check:

```yaml
Checks: >
  bugprone-*,
  -bugprone-easily-swappable-parameters,
  ...
```

After modifying `.clang-tidy`, re-run the analysis — no rebuild needed.
