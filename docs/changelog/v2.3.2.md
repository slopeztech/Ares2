# ARES v2.3.2 — Changelog

**Released:** 2026-05-22
**Previous version:** 2.3.1
**Type:** Patch — security hardening: first-boot credential generation, constant-time token comparison, WiFi flight disable, and related robustness fixes across the WiFi/API/config subsystem.

---

## Summary

This release resolves thirteen open security and quality findings ([C2], [C3], [C5], [H6], [H7], [H8], [H10], [H14], [M4], [M7], [M8], [M10], [M11]) that were identified during the structured security audit of the WiFi/API/config subsystem.

The most significant changes are:

- **First-boot credential generation ([C2], [C3])** — the factory default `ares1234` WiFi password and the absence of an API token are eliminated.  On the very first boot `DeviceConfig::provisionIfFirstBoot()` generates a 12-character alphanumeric WPA2 password and a 32-hex-character (128-bit) API bearer token via the ESP32 hardware TRNG (`esp_random()`).  Credentials are printed to USB-CDC and persisted to `/ares_device.json`.  From the second boot onward authentication is mandatory.

- **WiFi AP disabled in flight ([C5])** — `WIFI_DISABLE_IN_FLIGHT = true` was already defined in `config.h` but was never consumed.  `ApiServer::setMode()` now calls `wifi_.end()` when transitioning to `FLIGHT` and restarts the AP with `wifi_.begin()` on any transition back to `IDLE`/`TEST`/`ERROR`, guarded by `if constexpr`.

- **Constant-time token comparison ([M4])** — `checkToken()` previously iterated only `storedLen` bytes, leaving extra bytes in the provided token unseen.  The loop now runs `max(storedLen, providedLen)` iterations; bytes beyond the shorter string XOR against `'\0'`.

- **HTTP reason phrases ([H6])** — `sendJson()` was hardcoded to emit `"HTTP/1.1 NNN OK"` for every response code.  A switch now maps every used status code to its correct IANA reason phrase (`200 OK`, `201 Created`, `202 Accepted`, `204 No Content`, `400 Bad Request`, `401 Unauthorized`, `404 Not Found`, `405 Method Not Allowed`, `409 Conflict`, `413 Payload Too Large`, `414 URI Too Long`, `500 Internal Server Error`).

- **17 new unit tests** covering all fixed issues.  `api_routing` test suite grows from 10 to 27 tests.

### `src/sys/device_config/device_config.cpp`

| Location | Change |
|----------|--------|
| Platform wrappers | Added `platformRandom32()`, `fillHexRandom()`, `fillAlnumRandom()` — thin wrappers around `esp_random()` on Arduino and `rand()` on native test builds (`#ifdef ARDUINO`) |
| `provisionIfFirstBoot()` | Generates 12-char alphanumeric WPA2 password + 32-char hex API token; saves to flash; prints credentials via `LOG_I`; sets `firstBootProvisioned_ = true` |
| `wasFirstBootProvisioned()` | Returns `firstBootProvisioned_` |
| `checkToken()` | Loop extended to `max(storedLen, providedLen)` — **[M4]** |
| `toPublicJson()` | Early-return `0U` if `bufSize < 16` — **[M7]** |
| `validateWifiPass()` | Empty `wifi_password` field is now a no-op (preserves stored value) instead of reverting to `ares::WIFI_AP_PASSWORD`; error message updated accordingly — **[M10]** |

### `src/main.cpp`

| Location | Change |
|----------|--------|

### `src/api/api_server.cpp`

| Location | Change |
|----------|--------|

## Security findings resolved

| ID | Severity | Description |
|----|----------|-------------|
| [C2] | Critical | Factory WiFi password `ares1234` hardcoded; now replaced by per-device TRNG random at first boot |
| [C3] | Critical | API auth disabled by default; token now mandatory from first boot |
| [C5] | Critical | `WIFI_DISABLE_IN_FLIGHT` was dead code; WiFi AP now actually disabled in `FLIGHT` mode |
| [H6] | High | `sendJson()` emitted incorrect HTTP reason phrases |
| [H7] | High | PUT `/api/device/config` returned `200 OK` even when a WiFi password change required reboot; now `202 Accepted` |
| [H8] | High | `/api/scans/i2c` and `/api/scans/uart` were not blocked in flight mode |
| [H10] | High | `handleAbort()` returned 409 when mode had already left `FLIGHT` but engine was still `RUNNING` |
| [H14] | High | Double-`begin()` on WiFi AP and LittleFS panicked via `ARES_ASSERT`; now returns safe state |
| [M4] | Medium | `checkToken()` only checked `storedLen` bytes, leaving extra provided bytes unseen |
| [M7] | Medium | `toPublicJson()` could write past buffer end when `bufSize == 0` |
| [M8] | Medium | Oversized HTTP path silently truncated and routed as a shorter path |
| [M10] | Medium | Empty `wifi_password` in PATCH reverted to hardcoded default instead of preserving the stored value |
| [M11] | Medium | `sendError()` buffer overrun possible via long error message with fragile `ARES_ASSERT` |
