# Changelog — v2.6.5

**Date:** 2026-07-01  
**Type:** Performance / Observability (Minor)

---

## Summary

Version 2.6.5 starts a focused performance stabilization pass for the mission loop and logging path.
The first change is intentionally low-risk: we added optional timing instrumentation to isolate the
cost of GPS polling, radio polling, AMS tick execution, deferred I/O flushes, and storage writes.
The next step decouples the hot-path CSV append work from the mission tick by draining it on a
low-priority static FreeRTOS worker while keeping checkpoint writes on the explicit control path.
The profiling path remains disabled by default and does not change runtime behavior unless
explicitly enabled for lab measurements.

**Work completed so far:**
- ✅ Added `LOOP_TIMING_PROFILE_ENABLED` in `src/config.h` as an opt-in lab profiling flag.
- ✅ Instrumented the main loop to measure `gps->update()`, `radioDispatcher.poll()`, and `missionEngine.tick()`.
- ✅ Instrumented deferred AMS I/O flushes and LittleFS append/write paths.
- ✅ Instrumented LoRa TX latency to separate radio cost from AMS and storage cost.
- ✅ Documented the profiling flag in the AMS programming guide.
- ✅ Moved CSV append draining to a low-priority static worker so the mission tick no longer owns per-row LittleFS appends.
- ✅ Increased deferred append queue depth and added same-path append batching to reduce LittleFS call rate under 50 ms logging workloads.
- ✅ Fixed AMS state-directive bridge so unspecified wifi/api directives no longer force implicit enable, preventing AP re-enable races during FLIGHT transitions.
- ✅ Promoted IMU I2C1 bus from fixed conservative mode to a selectable safe/fast profile in `config.h` and enabled the 100 kHz fast profile by default, keeping a one-flag rollback path to 50 kHz for marginal wiring.
- ✅ Tuned BMP280 runtime profile for 50 ms logging workloads by moving register profile values to `config.h` and reducing sensor standby from 500 ms to 0.5 ms, preventing long plateaus of repeated barometer samples.
- ✅ Fixed static FreeRTOS task creation stack-depth units (AMS I/O worker, API task, LED task): `xTaskCreateStaticPinnedToCore` now receives depth in `StackType_t` entries instead of bytes, preventing out-of-bounds stack metadata and `ams_io` stack canary panics under load.
- ✅ Increased `TASK_STACK_SIZE_AMS_IO` from 4096 to 6144 bytes to add safety margin during deferred append batch drains.
- ✅ Hardened large log downloads (`GET /api/logs/:id`) against API task watchdog resets by feeding TWDT inside the chunk loop, yielding every few chunks, validating short socket writes, and increasing `DOWNLOAD_CHUNK_SIZE` to reduce per-chunk LittleFS overhead.

---

## Current Focus

The immediate goal for this version is to keep removing flash work from the hot path without
breaking checkpoint integrity or ARES timing rules. The current slice shifts the most frequent
LittleFS append cost off the tick; the remaining checkpoint path can be tightened next if profiling
shows it still dominates.

Expected follow-up areas:
- flash append and checkpoint write batching,
- suppression or rate limiting of overload WARN events,
- radio TX decoupling under congestion,
- sensor-bus timing improvements where hardware allows.

---

## Notes

- No functional mission logic has been changed by this first step.
- The profiling path remains disabled unless explicitly enabled in `src/config.h`.
- Future entries in this release should record one mitigation at a time so regressions are easy to
  isolate and roll back.
- Checkpoint writes still run through the explicit flush path; this release is reducing latency in
  stages so each change stays bounded and measurable.
