## Summary

This draft release captures the pending changes currently in the working tree and ready for the next commit:

- Added state-scoped AMS directives for `wifi.enable`, `wifi.disable`, `api.enable`, and `api.disable`.
- Wired the AMS runtime to apply state directives when entering a state, so WiFi/API behaviour can be overridden per mission phase.
- Added the configurable `WIFI_DISABLE_IN_FLIGHT` policy in `src/config.h`; the default is now `false` so the AP stays available during the mission unless you explicitly opt into flight-time shutdown.

- Extended the offline AMS validator (`scripts/ams_parser.py`) so the new directives are accepted before upload.
- Enabled serial logging for the WiFi password and API token during diagnostics.
- Updated the WiFi/API documentation to describe the new default policy and the AMS override path.

These changes are implementation-focused and keep the existing AMS execution model intact while adding per-state WiFi/API policy control. The current master-pipeline run is green overall (6 pass, 1 warning, 0 fail), with only two warning-only clang-tidy threshold warnings remaining in the runtime and main loop paths.

## Changes

### 1. AMS state directives

The AMS engine now understands state-scoped directives for WiFi and API control:

- `wifi.enable`
- `wifi.disable`
- `api.enable`
- `api.disable`

This enables mission scripts to decide, per state, whether the WiFi AP and the REST API surface should stay active.

### 2. Runtime callback wiring

The mission runtime now invokes a state-directive callback when entering a state that declares one of these directives. The callback decides how the live runtime should react to the directive pair.

### 3. API server / runtime integration

The WiFi AP can now be enabled or disabled independently of the operating-mode transition logic, which allows AMS states to override the default flight-time policy for a specific state.

The global default is now controlled by `WIFI_DISABLE_IN_FLIGHT` in `src/config.h`. With the default value set to `false`, the AP remains up during mission unless you choose to force a hard shutdown on `FLIGHT` mode. AMS directives still win on a per-state basis when they are present.

### 4. Offline AMS validation

The Python AMS validator now accepts the new directives, so the same syntax is validated locally before flashing the board.

### 5. Diagnostics helpers

The serial password/token debug logging switches are now enabled in `src/config.h`, which improves live troubleshooting during mission tests.
