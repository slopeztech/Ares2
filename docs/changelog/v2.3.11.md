# ARES v2.3.11 — Changelog

**Released:** 2026-05-31
**Previous version:** 2.3.10
**Type:** Patch — radio dispatcher race hardening (P1-2, P1-4); dxlr03 magic-number cleanup (P3-1).

---

## Summary

This release closes three code-review findings from the post-v2.3.9 audit.

**P1-2** — `rxBuf_` / `rxLen_` in `RadioDispatcher` were shared mutable state
accessible from `poll()` without an ownership contract.  The design intention is
that `poll()` is always called from a single, dedicated FreeRTOS task, but
nothing enforced this.  A task handle is now latched on the first `poll()` call
(`pollTask_`) and `ARES_REQUIRE` aborts the system immediately in release if any
other task ever calls `poll()`.  No mutex overhead is added because the single-
task contract is the correct model; `ARES_REQUIRE` is never elided
(`esp_system_abort()` on violation).

**P1-4** — `proto::encode()` previously returned `uint16_t`: both `0` on error
and `0` for a legitimately zero-length encode were indistinguishable, so every
caller had to guess.  The signature is changed to
`bool encode(const Frame& frame, uint8_t* buf, uint16_t bufLen, uint16_t& outLen)`:
`false` is an unambiguous error, `true` with `outLen` is the valid encoded
length.  All nine call sites in `radio_dispatcher.cpp`, one in
`mission_script_engine_telemetry.cpp`, and all test helpers in
`test_radio_protocol/` are updated accordingly.

**P3-1** — `dxlr03_driver.cpp` contained two magic numbers with no documented
rationale:

- `512` in `begin()` — the RX-FIFO flush bound.  A new project constant
  `LORA_UART_RX_BUF_BYTES = 512U` is added to `config.h`, mirrored as the
  class constant `UART_RX_BUF_BYTES` in the driver header, and used in place of
  the literal.
- `2000` in `waitReady()` — a max-polls cap that was inconsistent with the
  already-named `AUX_TIMEOUT_MS = 50 ms`.  Replaced by the compile-time
  expression `static_cast<uint16_t>(AUX_TIMEOUT_MS / POLL_PERIOD_MS)` so the
  cap always tracks the timeout constant.
