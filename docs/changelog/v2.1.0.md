# Changelog — v2.1.0

**Date:** 2026-05-17
**Type:** Feature + Breaking Change

---

## Summary

Version 2.1.0 introduces the `pulse.channel` declaration syntax (AMS-4.18),
extends PULSE support from 2 to **4 channels** (A/B/C/D), renames the pulse
GPIO constants to a hardware-agnostic scheme (`PIN_PULSE_A/B/C/D`), and
establishes the official project versioning policy.

---

## New Features

### AMS-4.18 — `pulse.channel` declaration directive

A new metadata-section directive declares which physical pulse channels a
mission script intends to use, and optionally assigns a human-readable label:

```ams
pulse.channel A as DROGUE
pulse.channel B as MAIN
pulse.channel C as STAGE_SEP
```

- Channels A, B, C, D map to GPIO pins `PIN_PULSE_A`, `PIN_PULSE_B`,
  `PIN_PULSE_C`, and `PIN_PULSE_D` respectively.
- The optional `as LABEL` clause lets `PULSE.fire` reference channels by name
  instead of bare letter (e.g. `PULSE.fire DROGUE`).
- Labels must be alphanumeric + underscore, max 15 characters, and unique.
- A declared-but-never-used channel produces a parser warning (`LOG_W`).

### 4-channel PULSE support

The pulse subsystem now supports four independent channels:

| Channel | Config pin    | GPIO |
|---------|---------------|------|
| A       | `PIN_PULSE_A` | 4    |
| B       | `PIN_PULSE_B` | 15   |
| C       | `PIN_PULSE_C` | 16   |
| D       | `PIN_PULSE_D` | 17   |

`AMS_MAX_PULSE_ACTIONS` raised from 2 to 4 (one per channel).

### Hardware-agnostic GPIO naming

GPIO constants renamed from `PIN_DROGUE` / `PIN_MAIN` to `PIN_PULSE_A` /
`PIN_PULSE_B`. Purpose is expressed only in the `.ams` script via the `as LABEL`
clause — the firmware layer has no concept of "drogue" or "main".

### Status bits for channels C and D

The HK telemetry frames include a status byte that indicates which pulse
channels were fired during the mission.  In v2.1.0 bits 6 and 7 (previously
reserved) are assigned to channels C and D:

```
Status byte (8 bits):

  bit 7    bit 6    bit 5    bit 4    bit 3    bit 2    bit 1    bit 0
┌────────┬────────┬────────┬────────┬────────┬────────┬────────┬────────┐
│  CH_D  │  CH_C  │   —    │   —    │   —    │   —    │  CH_B  │  CH_A  │
│ 0x80   │ 0x40   │        │        │        │        │ 0x02   │ 0x01   │
└────────┴────────┴────────┴────────┴────────┴────────┴────────┴────────┘
  new       new                                          prev     prev
```

A bit set to **1** means that channel was fired at some point during script
execution.  Bits 2–5 are reserved (always 0).

| Constant               | Bit | Hex    | Meaning                    |
|------------------------|-----|--------|----------------------------|
| `STATUS_PULSE_A_FIRED` | 0   | `0x01` | Channel A fired            |
| `STATUS_PULSE_B_FIRED` | 1   | `0x02` | Channel B fired            |
| `STATUS_PULSE_C_FIRED` | 6   | `0x40` | Channel C fired *(new)*    |
| `STATUS_PULSE_D_FIRED` | 7   | `0x80` | Channel D fired *(new)*    |

Bits 6 and 7 replace `STATUS_RESERVED` (`0xC0`), which previously occupied
those positions with no defined meaning.

## Breaking Changes

### `PULSE.fire` requires a prior `pulse.channel` declaration

Scripts that use `PULSE.fire A` without a matching `pulse.channel A` directive
will fail at `activate()` with:

```
PULSE.fire: channel used but not declared with pulse.channel
```

**Migration:**

```ams
# Before v2.1.0 (no longer valid):
state DEPLOY:
  on_enter:
    PULSE.fire A

# v2.1.0+:
pulse.channel A as DROGUE

state DEPLOY:
  on_enter:
    PULSE.fire DROGUE
```

### `PulseDriver` constructor signature changed

The `PulseDriver` constructor now takes an array of `PulseDriver::ChannelConfig`
structs:

```cpp
static const PulseDriver::ChannelConfig kPulseChannels[PulseChannel::COUNT] = {
    { ares::PIN_PULSE_A, PulseDriver::kNoPinAssigned },
    { ares::PIN_PULSE_B, PulseDriver::kNoPinAssigned },
    { ares::PIN_PULSE_C, PulseDriver::kNoPinAssigned },
    { ares::PIN_PULSE_D, PulseDriver::kNoPinAssigned },
};
static PulseDriver pulse(kPulseChannels);
```

## New Features (AMS-4.19 Pulse Safety System)

### AMS-4.19 — Five-gate pyrotechnic safety system

A complete safety framework has been added for pyrotechnic pulse channels.
Five independent gates must all pass before `PULSE.fire` releases energy.

#### AMS-4.19.1 — Two-phase arm–fire (`PULSE.arm`)

A new `PULSE.arm <channel>` action (inside `on_enter` blocks) sets an arm
token. Any channel referenced by `PULSE.arm` anywhere in the script requires
that token to be present at the moment `PULSE.fire` is evaluated. The token is
consumed on successful fire.

```ams
state ARM:
  on_enter:
    PULSE.arm DROGUE

state FIRE:
  on_enter:
    PULSE.fire DROGUE   // only fires if arm token is set
```

#### AMS-4.19.2 — Minimum altitude (`pulse.min_altitude`)

```ams
pulse.min_altitude 300    // blocked below 300 m ASL
```

Range: **[1, 50 000]** m. Evaluated against the first declared barometric driver.

#### AMS-4.19.3 — Arm token timeout (`pulse.arm_timeout`)

```ams
pulse.arm_timeout 5000    // arm token expires after 5 s
```

Range: **[500, 300 000]** ms. Expired tokens are cleared on each `engine.tick()`.

#### AMS-4.19.4 — Bridgewire continuity check (`pulse.require_continuity`)

```ams
pulse.require_continuity A    // fire blocked if ch A bridgewire is open
```

Validated at **parse time**: if no hardware continuity pin is wired for the
channel (`PIN_PULSE_x_CONT == PIN_NO_CONT`), `activate()` fails immediately.
At runtime, the GPIO continuity pin is sampled on each `PULSE.fire`.

#### AMS-4.19.5 — Post-arm safe delay (`pulse.safe_delay`)

```ams
pulse.safe_delay 2000    // no fire for 2 s after engine.arm()
```

Range: **[100, 60 000]** ms. Measured from the instant `engine.arm()` is called.

---

### Hardware continuity pin constants (`src/config.h`)

Four new constants define the hardware continuity input pins per channel:

```cpp
constexpr uint8_t PIN_NO_CONT      = 0xFFU;   // sentinel — no pin wired
constexpr uint8_t PIN_PULSE_A_CONT = PIN_NO_CONT;
constexpr uint8_t PIN_PULSE_B_CONT = PIN_NO_CONT;
constexpr uint8_t PIN_PULSE_C_CONT = PIN_NO_CONT;
constexpr uint8_t PIN_PULSE_D_CONT = PIN_NO_CONT;
```

Assign actual GPIO numbers to enable `pulse.require_continuity` in scripts.

### GPIO output order fix (`src/drivers/pulse/pulse_driver.cpp`)

`begin()` now calls `digitalWrite(LOW)` before `pinMode(OUTPUT)` for each
channel, eliminating the glitch window during which a channel could
momentarily drive HIGH before being configured.

## Bug Fixes and Reliability Improvements

### Bug: per-channel fired-status flags not reset on `deactivate()`

`deactivateLocked()` reset the AMS-4.19 arm-token state (`pulseArmed_[]`,
`pulseArmedMs_[]`) but did **not** reset the four fired-status flags
(`pulseAFired_`, `pulseBFired_`, `pulseCFired_`, `pulseDFired_`). After
deactivating and reactivating the same engine, `getStatusBits()` would still
report channels as fired from the prior session.

**Fix:** all four flags are now explicitly reset to `false` in `deactivateLocked()`.

### Bug: `pulse.safe_delay` could be bypassed at boot when `millis64() == 0`

`arm()` stored the activation timestamp as `activationMs_ = millis64()`. The
`pulse.safe_delay` gate condition is `activationMs_ > 0U`; if `arm()` happened
to be called at the exact instant `millis64()` returned `0` (e.g. the very
first millisecond after power-on), the gate was permanently disabled and the
safe-delay constraint was ignored for that session.

**Fix:** `activationMs_` is now written as `max(millis64(), 1U)`, guaranteeing
the safe-delay gate is always active once `pulse.safe_delay` is declared.

### Improvement: duplicate scalar pulse directives now rejected

Previously, writing `pulse.safe_delay`, `pulse.arm_timeout`, or
`pulse.min_altitude` more than once in a script was silently accepted and the
last occurrence would win. The AMS specification (AMS-4.19.2, AMS-4.19.3,
AMS-4.19.5) already stated that only one occurrence is allowed; the parser now
enforces this:

```
pulse.safe_delay: duplicate directive (only one allowed per script)
pulse.arm_timeout: duplicate directive (only one allowed per script)
pulse.min_altitude: duplicate directive (only one allowed per script)
```

### Improvement: `pulse.min_altitude` behaviour when no baro driver is configurable

When a script declares `pulse.min_altitude` but the engine is constructed
without any barometric driver (`baroCount = 0`), the altitude gate now honours
the new `pulse.no_baro_policy` directive (AMS-4.19.6):

- **`pulse.no_baro_policy block`** (default when the directive is absent):
  fire is suppressed and a `W`-level warning is emitted:
  `PULSE ch=N blocked: pulse.min_altitude set but no baro driver registered`
- **`pulse.no_baro_policy allow`**: the altitude gate is skipped entirely and
  an `I`-level note is logged; all other safety gates still apply.
