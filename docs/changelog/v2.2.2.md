# ARES v2.2.2 — Changelog

**Release date:** 2026-05-18
**Branch:** `feature/coverage-improvements`
**Type:** Patch — radio protocol safety fix (FIRE_PULSE_C / FIRE_PULSE_D)

---

## Summary

Resolves **[C4]** from the static analysis backlog: `FIRE_PULSE_C` and
`FIRE_PULSE_D` were silently falling through to `UNKNOWN_COMMAND` in the
radio dispatcher despite the HAL, engine flags, and `PulseChannel` enum all
supporting four channels since v2.1.0.

- `ares_radio_protocol.h` — `CommandId::FIRE_PULSE_C = 0x07`,
  `CommandId::FIRE_PULSE_D = 0x08` added to the enum.
- `radio_dispatcher.cpp` — three enforcement points updated:
  `isCritical` guard, `executeCommand` switch, `commandPriority`.
- 5 new unit tests added to `test_dispatcher` (total 50 tests).
- Gap-commandId test updated: `0x07` → `0x09` (0x07 is now FIRE_PULSE_C).

No firmware behaviour changes for existing radio ground stations that only
use channels A and B.

---

## Changes

### [C4] FIRE_PULSE_C / FIRE_PULSE_D — radio dispatcher

**Root cause:** The `CommandId` enum only declared `FIRE_PULSE_A = 0x03` and
`FIRE_PULSE_B = 0x04`.  Any COMMAND frame carrying `0x07` or `0x08` passed
the `[FIRST, LAST]` range check (FIRST=0x01, LAST=FACTORY_RESET=0x24) but hit
the `default:` branch in the `executeCommand` switch, returning
`UNKNOWN_COMMAND` without any pyrotechnic action — and without any clear
error to the ground operator.

Additionally, `isCritical` and `commandPriority` did not include C/D.  A
hypothetical caller that injected frames for C/D without `FLAG_PRIORITY` would
have bypassed the critical-command guard entirely.

**Fix:**

| Point | Before | After |
|-------|--------|-------|
| `CommandId` enum | FIRE_PULSE_A/B only | + FIRE_PULSE_C=0x07, FIRE_PULSE_D=0x08 |
| `executeCommand` switch | `case FIRE_PULSE_A: case FIRE_PULSE_B:` | + `FIRE_PULSE_C` + `FIRE_PULSE_D`; `ch` mapping uses `if/else if` chain |
| `isCritical` predicate | A, B, ABORT, FACTORY_RESET | + C, D |
| `commandPriority()` | A, B → kCriticalPriority | + C, D → kCriticalPriority |

`notifyPulseFired()` already handled channels 2 (CH_C) and 3 (CH_D) since
v2.1.0 — no engine changes required.

## Protocol wire compatibility

`FIRE_PULSE_C = 0x07` and `FIRE_PULSE_D = 0x08` were previously unassigned in
the wire protocol.  Ground stations that only transmit A/B commands are
unaffected.  Ground stations wishing to use C/D must send
`FLAG_PRIORITY | commandId=0x07/0x08` in the COMMAND frame.

---

## Security note

Per **APUS-7.3**, both new commands require `FLAG_PRIORITY`.  Any COMMAND
frame for C/D without that flag is rejected with `PRECONDITION_FAIL` before
the `executeCommand` switch is entered.  This prevents accidental or
unauthenticated actuation (see also **[C1]** for the broader HMAC roadmap).
