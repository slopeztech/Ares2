# ARES v2.0.0 — Changelog

**Released:** 2026-05-16  
**Previous version:** 0.1.0  
**Type:** Major release — significant architectural changes, breaking API changes.

---

## Summary

v2.0.0 represents the first major structured release of ARES after the initial prototype (v0.1.0). The primary changes are a full refactor of the AMS engine into modular files, the introduction of REST API authentication, and a comprehensive documentation overhaul aligned with the hardware and standards actually in use.

---

## Changed

### AMS Engine — Refactor into modules

The monolithic AMS implementation was split from two large files into eight focused modules to improve readability, maintainability, and test isolation:

| Old file | New modules |
|----------|-------------|
| `mission_script_engine_parser.cpp` (≈4013 lines) | `ams_parser_core.cpp`, `ams_parser_directives.cpp`, `ams_parser_expressions.cpp`, `ams_parser_states.cpp`, `ams_parser_validation.cpp` |
| `mission_script_engine.cpp` (≈2858 lines) | `ams_engine_runtime.cpp`, `ams_engine_telemetry.cpp`, `ams_engine_guards.cpp` |

No AMS language behaviour changed. All existing `.ams` scripts remain compatible.

### Radio — Hardware update

- **Module changed** from `DX-LR03-433T30D` to `DX-LR03-900`.
- **Frequency changed** from 433.125 MHz to **868 MHz**.
- Radio layer remains transport-agnostic; only `DxLr03Driver` and `config.h` channel settings change.

### REST API — Authentication

Added `X-ARES-Token` header support for all REST API endpoints:

- Clients must send `X-ARES-Token: <token>` on requests to protected routes.
- Token is configurable at runtime via `PUT /api/device/config` and persisted to LittleFS.
- Default token: empty string (no auth, backward-compatible with v0.1.0 ground station tools). Ground station tools that do not send the header will work against a device with the default empty token.
- Python TUI scripts (`scripts/ares_api_tui.py`, `scripts/ares_fs_tui.py`) updated with `--token` CLI argument.

### Pin naming — Pyro → Pulse

GPIO 4 and GPIO 15 are now consistently referred to as **Pulse DROGUE** and **Pulse MAIN** across all documentation and in `config.h` (`PIN_DROGUE`, `PIN_MAIN`). The previous name "Pyro" implied a specific actuator type; "Pulse" is hardware-neutral.

### WiFi REST API — Configurability

SSID, password, `X-ARES-Token`, and CORS settings are now all runtime-configurable via `PUT /api/device/config` and persisted to LittleFS. Prior to this release only `config.h` compile-time values existed.

## Fixed

### MISRA-5.6 violations (×4)

Four MISRA C++ 5.6 violations (duplicate `using` declarations across translation units in the same namespace) were eliminated by removing redundant declarations and qualifying types with the `detail::` namespace prefix.

### Unity sim linker error

The `[env:sim]` PlatformIO environment `build_src_filter` was incomplete, causing linker failures when running `pio test -e sim`. The filter was expanded to include all AMS engine modules introduced by the refactor.

## Added

### CI Master pipeline (`CI: Master pipeline` VS Code task)

A unified quality pipeline was introduced that runs all seven verification stages in sequence and produces a single `master.txt` report:

1. Build (`pio run`)
2. Compile DB (`pio run -t compiledb`)
3. MISRA Cppcheck (with MISRA C 2012 addon)
4. Cppcheck Practical (XML report, no MISRA)
5. clang-tidy (LLVM 22)
6. Unity tests (`pio test -e sim`) — 7 test suites
7. Coverage (gcov)

Reports are archived under `evidence/<date>/` and mirrored to `evidence/latest/`.

### Version control (changelog)

This file is the first entry in `docs/changelog/`. Going forward, each release produces a `vX.Y.Z.md` file in this folder documenting all changes.

The authoritative version number lives in:

```cpp
// src/config.h
#define ARES_VERSION_STRING "2.0.0"
```

When bumping the version, update `ARES_VERSION_STRING` first, then update:
- `README.md` — badge, hardware table row, serial log example
- `docs/requirements/SRS.sdoc` — `TITLE` and `VERSION` fields
- `docs/api/wifi_api_endpoints.md` — `GET /api/status` response example
- Create a new `docs/changelog/vX.Y.Z.md`

## Documentation overhaul

The `README.md` was substantially rewritten to reflect v2.0.0 reality:

- **What is ARES?** — rewritten to lead with AMS as the central concept.
- **Current Drivers** (previously *Current Peripherals*) — driver-per-row table with HAL contract column; IMU naming de-emphasised from primary/secondary to driver role.
- **Firmware Architecture** — expanded with per-module responsibility breakdown and concurrency model.
- **Runtime Flow** — rewritten as a numbered lifecycle without hard-coding chip names.
- **AMS section** — expanded with all compile-time limits from `config.h`.
- **Radio** — updated to DX-LR03-900 / 868 MHz.
- **WiFi REST API** — added configurability note.
- **Coding Standards** — all 11 standards documented with rationale (previously only 3 were explained).
- **Requirements Management** — added "Work in progress" banner.
- **Testing** — full CI master pipeline stages and all 7 Unity test suites documented.
- **Badges** — expanded from 4 to 8 (added Tests, Language, Radio, License).

## Known limitations / not in this release

- SRS requirement coverage is partial. Traceability links from source code to SRS-NNNN UIDs exist for the AMS and platform subsystems but are not yet complete for the API, storage, and RTOS sections.
- No hardware-in-the-loop (HIL) test bench. All automated tests run on the host `sim` environment with HAL stubs.
- The `X-ARES-Token` mechanism provides application-layer access control but the transport is HTTP (not HTTPS). This is acceptable on the isolated WiFi AP but should be noted for any future remote-access use case.
