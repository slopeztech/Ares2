# ARES v2.3.13 — Changelog

**Released:** 2026-05-31
**Previous version:** 2.3.12
**Type:** Patch — P0-1 `tick()` single-exit flush fix; P3-1 `uint64_t` elapsed; 4 new SITL flush tests.

---

## Summary

This release closes **P0-1** (critical) from the post-v2.3.12 audit: the AMS
engine `tick()` method contained five `return` statements inside its mutex
section that bypassed `flushPendingIoUnlocked()`, silently discarding staged
checkpoint writes and abort-row appends.  The method is restructured to a
single exit point so the flush is unconditionally reached on every code path.

A related cosmetic finding (**P3-1**) is also fixed: the elapsed-time variable
in the fallback block was typed `uint32_t`, truncating the `uint64_t` timestamp
subtraction.  It is now `uint64_t` consistent with `checkOnTimeoutLocked`.

Four new SITL integration tests in `test_ams_checkpoint_flush.cpp` verify that
each of the previously-skipped paths now correctly persists state.

---

## Changes

**P0-1 — `tick()` skips `flushPendingIoUnlocked()` on all transition exit paths**

`mission_script_engine_runtime.cpp`: The five interior `return` statements that
exited `tick()` while still inside the `ScopedLock` scope — reached via
`evaluateTransitionAndMaybeEnterLocked()`, `checkOnTimeoutLocked()`, the
fallback block, and the ABORT force-deactivation branch — have been removed.
The logic is restructured into nested boolean guards (`transitioned`,
`didFallback`) so that execution always falls through to the closing brace of
the mutex scope and then to `flushPendingIoUnlocked()`.

Three early `return` statements remain.  All three occur *before* any staging
takes place and are therefore safe:
- `!guard.acquired()` — mutex not acquired; nothing staged.
- `!running_ || status_ != RUNNING || !executionEnabled_` — engine idle/paused.
- `currentState_ >= stateCount` — program table corrupt; no staging attempted.

Consequences of the bug (before fix): every state-transition checkpoint written
by `enterStateLocked()` (called on TC transitions, fallback transitions, and
on_timeout transitions) was staged in `pendingCheckpoint_` but never written to
LittleFS.  Abort audit rows queued by `writeAbortMarkerLocked()` were similarly
lost.  A power cycle after any transition would have left the checkpoint file
stale (pointing at the pre-transition state), causing incorrect mission resume.

**P3-1 — `uint32_t` elapsed truncation in fallback block**

The local variable `elapsed` used to compare `nowMs - stateEnterMs_` against
`state.fallbackAfterMs` was declared as `uint32_t`.  Because both operands are
`uint64_t`, the subtraction result was silently truncated.  For missions longer
than ~49 days this would cause the fallback to fire spuriously; while benign in
practice, it was semantically inconsistent with the identical computation in
`checkOnTimeoutLocked`, which already uses `uint64_t`.  Fixed to `uint64_t`.

## New tests

**`test/test_ams_integration/test_ams_checkpoint_flush.cpp`** (4 tests)

| Test | Exit path exercised |
|------|---------------------|
| `test_checkpoint_written_on_tc_transition` | `evaluateTransitionAndMaybeEnterLocked()` returns `true` (WAIT→FLIGHT via LAUNCH TC) — checkpoint for stateIdx=1 written |
| `test_checkpoint_written_on_fallback_transition` | Fallback fires (FLIGHT→SAFE after 2001 ms) — checkpoint for stateIdx=2 written |
| `test_checkpoint_written_on_timeout` | `checkOnTimeoutLocked()` returns `true` (HOLD→TIMEOUT_WIN at 3001 ms) — checkpoint for stateIdx=2 written |
| `test_abort_row_flushed_on_unhandled_abort` | ABORT TC with no handler — `writeAbortMarkerLocked()` abort row flushed via `appendFile` |

All 4 tests pass.  Total test suite: **183 tests, 183 passed**.
