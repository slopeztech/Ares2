# ARES v2.3.12 — Changelog

**Released:** 2026-05-31
**Previous version:** 2.3.11
**Type:** Patch — SIOF fix main.cpp (P1-3); checkpoint/telemetry truncation guards (P2-1, P2-4); fill_n/kCrcSuffix cleanup (P2-5, P3-2); API buffer single-task contract (P2-2).

---

## Summary

This release closes seven code-review findings from the post-v2.3.11 audit plus
two PR #28 review comments.

**P1-3 — SIOF in `main.cpp` (Static Initialisation Order Fiasco)**  
`Bmp280Driver` and `ApiServer` were previously constructed as file-scope objects
with non-trivial constructors, making them susceptible to Static Initialisation
Order Fiasco across translation units.  Both are now deferred via placement new
into static aligned buffers (`s_baroBuf` / `s_apiBuf`), constructed explicitly
inside `setup()` after all dependencies are ready.  The mutable registries that
pointed into them (`kBaroIfaces`, `kBaroDrivers`) are renamed to `s_baroIfaces`
/ `s_baroDrivers` to reflect their mutability (patched in `setup()`, not const
at link time).  A `startApiServer()` helper is extracted for clang-tidy
function-size compliance (`setup()` → 79 lines, threshold 80); it also
satisfies the PR #28 null-safety review comment: `pApiServer` is only published
after a successful `begin()`.  A null guard in `loop()` prevents TWDT
registration before the server is alive.

**P2-1 — Checkpoint buffer truncation**  
`mission_script_engine_checkpoint.cpp`: after `buf[bytesRead] = '\0'`, a guard
now detects when the checkpoint file was too large to fit in the 512-byte read
buffer (`bytesRead >= sizeof(buf) - 1U`).  On detection: `LOG_W` + discard via
`clearResumePointLocked()` instead of silently resuming from a truncated state.

**P2-2 — API response buffers single-task contract**  
`mission_handler.cpp` and `storage_handler.cpp`: the existing one-sentence
comment is replaced by a structured block that names the owning task
(`ApiServer::taskFn`, REST-13/RTOS-2), states the invariant (one request at a
time, no concurrent access), and explicitly warns that any future caller from a
second task must serialize via the ApiServer config mutex.

**P2-4 — Telemetry snprintf truncation detection**  
`mission_script_engine_telemetry.cpp`: three `snprintf` loops (slot header in
`appendLogReportSlotLocked`, slot data row in `appendLogReportSlotLocked`, and
legacy header in `writeLogHeaderIfNeededLocked`) previously checked only
`n <= 0`, missing the `n >= remaining` truncation case.  Each loop now uses a
counter variable declared outside the loop body; an `ARES_ASSERT` after each
loop fires in debug builds if any field was silently dropped.  In release builds
the CRC8 field at row end already signals ground-station parsers that a row is
incomplete (AMS-4.3.2).

**P2-5 — `memset` → `std::fill_n` for slot timestamp arrays**  
`mission_script_engine.cpp`: `memset(lastHkSlotMs_, 0, sizeof(…))` and
`memset(lastLogSlotMs_, 0, sizeof(…))` replaced with
`std::fill_n(…, ares::AMS_MAX_HK_SLOTS, uint64_t{0U})`.  The element type and
count are now the same symbolic constants used in the array declaration, so a
future type change produces a compile error instead of partial/incorrect zeroing.

**P3-2 — `kCrcSuffix` constant in checkpoint abort marker**  
`mission_script_engine_checkpoint.cpp`: the magic literal `5U` used twice in
`writeAbortMarkerLocked` (once as `snprintf` size limit, once as the residual-
space guard) is replaced by `static constexpr size_t kCrcSuffix = 5U;` with a
comment that documents its composition (`","` + 2 hex + `"\n"` + NUL).
