## Summary

Version 2.1.1 addresses two independent bugs in the AMS metadata parser, one
diagnostic cosmetic issue, and three documentation/consistency improvements:

1. **AMS-4.18.6** (`pulse.channel`) â€” A malformed declaration such as
   `pulse.channel A FOO` was silently accepted; the unrecognised trailing token
   was discarded and the label defaulted to `"A"`.  The parser is now strict:
   any non-whitespace token after the channel letter that is not the `as LABEL`
   clause is a hard parse error.

2. **AMS-4.19.6** (`pulse.no_baro_policy`) â€” Unlike every other pulse safety
   directive, `pulse.no_baro_policy` had no duplicate-declaration guard.  A
   script with two contradictory lines was silently accepted with last-one-wins
   semantics.  The parser now rejects any second occurrence with an explicit
   error.

3. **`PulseDriver` ready log** â€” Unassigned channels (`kNoPinAssigned`) were
   printed as `GPIO255` in the `begin()` ready message.  They now print `--`,
   making the wiring map immediately readable in the boot log.

4. **`pulse.min_altitude` MSL/AGL alignment** â€” Three comments contradictorily
   described the altitude reference as AGL when the barometric driver interface
   (`BaroReading::altitudeM`) and the AMS standard both define it as MSL.  All
   four affected locations now consistently say MSL.

5. **`arm()` checkpoint timestamp consistency** â€” `saveResumePointLocked` was
   called with a second `millis64()` instead of the already-captured `nowMs`;
   elapsed-time fields in the checkpoint are now derived from the same snapshot
   as `activationMs_`.  A design comment clarifies that `activationMs_` is not
   persisted and that `pulse.safe_delay` resets on every `arm()` call.

6. **Stale test timing comments** â€” Two inline comments in
   `test_ams_pulse_safety.cpp` described pre-`armAndTickAt` timing semantics
   (`activationMs_ = ~0`, `arm() at t=0`); both updated to match the helper's
   actual contract.

## Bug Fixes

### AMS-4.18.6 â€” Strict trailing-token check in `pulse.channel`

**Affected component:** `parsePulseChannelLabelLocked`  
**File:** `src/ams/mission_script_engine_parser_meta.cpp`

#### Problem

When `parsePulseChannelLabelLocked` did not find the substring `" as "` in the
remainder of the directive, it unconditionally fell through to the no-alias
branch and used the channel letter as the label, silently discarding any
trailing content:

```ams
# v2.1.0 behaviour â€” silently accepted, FOO discarded, label = "A"
pulse.channel A FOO
```

This was inconsistent with AMS-4.18.6, which already listed
`"unexpected characters after channel letter"` as a parse error for the case
where the channel letter was followed by a non-space character.  The analogous
case with a space-separated non-`as` token was not covered.

#### Fix

In the `else` branch of `parsePulseChannelLabelLocked`, the function now
verifies that the remainder of the line (`rest + 1`) contains only trailing
whitespace before assigning the default label.  If non-whitespace content is
present (and no `" as "` was matched), it calls `setErrorLocked` with:

```
pulse.channel: unexpected token after channel letter; expected end-of-line or 'as LABEL'
```

Valid inputs are unaffected:

```ams
pulse.channel A          # valid â€” no suffix
pulse.channel A as MAIN  # valid â€” 'as LABEL' clause
```

#### Standard update

`docs/standards/ams_standard.md` Â§ AMS-4.18.6 now lists two distinct error
rows for malformed content after the channel letter:

| Condition | Error message |
|-----------|---------------|
| Non-space character immediately after channel letter (e.g. `pulse.channel AX`) | `pulse.channel: unexpected characters after channel letter` |
| Space-separated token that is not `as` (e.g. `pulse.channel A FOO`) | `pulse.channel: unexpected token after channel letter; expected end-of-line or 'as LABEL'` |

### AMS-4.19.6 â€” Duplicate `pulse.no_baro_policy` directive now rejected

**Affected component:** `parsePulseNoBaroPolicyLineLocked`  
**File:** `src/ams/mission_script_engine_parser_meta.cpp`

#### Problem

All other pulse safety directives (`pulse.min_altitude`, `pulse.safe_delay`,
`pulse.arm_timeout`) guard against duplicate declarations and fail the parse
with an explicit error.  `pulse.no_baro_policy` had no such guard: a script
with two contradictory declarations was silently accepted with last-one-wins
semantics:

```ams
# v2.1.0 behaviour â€” silently accepted; effective policy = block
pulse.no_baro_policy allow
pulse.no_baro_policy block
```

This was inconsistent with AMS-4.19 policy that each safety directive may
appear at most once per script.

#### Fix

Added a `bool pulseHasNoBaroPolicy` field to `ProgramData` (analogous to
`pulseHasMinAlt`).  `parsePulseNoBaroPolicyLineLocked` now checks this flag
before committing the directive and calls `setErrorLocked` with:

```
pulse.no_baro_policy: duplicate directive (only one allowed per script)
```

The field is set to `true` on the first successful parse, so any subsequent
occurrence is rejected.

#### Standard update

`docs/standards/ams_standard.md` Â§ AMS-4.19.6 error conditions table now
includes:

| Condition | Error message |
|-----------|---------------|
| Duplicate `pulse.no_baro_policy` directive | `pulse.no_baro_policy: duplicate directive (only one allowed per script)` |

### `PulseDriver::begin()` â€” Suppress misleading `GPIO255` in ready log

**Affected component:** `PulseDriver::begin()`  
**File:** `src/drivers/pulse/pulse_driver.cpp`

#### Problem

The ready log message unconditionally printed the raw `firePin` value for all
four channels as `GPIO%u`.  Channels not wired on a given board have their
`firePin` set to `kNoPinAssigned` (0xFF), so the message read:

```
ready (a=GPIO4 b=GPIO5 c=GPIO255 d=GPIO255)
```

The value `GPIO255` is meaningless and could cause an operator to mistake an
unassigned channel for a hardware fault.

#### Fix

A small `pinStr` lambda (block-scoped, four stack buffers of 8 bytes each)
formats each channel independently: wired channels keep the `GPIOnn` form;
unassigned channels (`firePin == kNoPinAssigned`) print `--`.  The same ready
message on a two-channel board now reads:

```
ready (a=GPIO4 b=GPIO5 c=-- d=--)
```

## Improvements

### `arm()` â€” Checkpoint timestamp consistency and `pulse.safe_delay` design note

**Affected component:** `MissionScriptEngine::arm()`  
**File:** `src/ams/mission_script_engine.cpp`

#### Problem

In `arm()`, `activationMs_` was stamped from a captured `nowMs`, but the
immediately following `saveResumePointLocked` call used a second `millis64()`
invocation.  The elapsed-time fields baked into the checkpoint (`stateElapsed`,
`hkElapsed`, `logElapsed`) were therefore derived from a slightly different
timestamp than `activationMs_`.

Additionally, it was undocumented at the assignment site that `activationMs_`
is intentionally absent from the checkpoint format and reset to 0 by
`deactivateLocked()`, leaving the intended lifetime of `pulse.safe_delay`
across power cycles ambiguous.

#### Fix

`saveResumePointLocked` now receives the already-captured `nowMs` instead of a
fresh `millis64()` call.  A design comment was added explaining:

- `activationMs_` is **not** serialised by `buildCheckpointRecordLocked`.
- `deactivateLocked()` explicitly resets it to `0U` on every deactivation.
- Therefore `pulse.safe_delay` is always measured from the current `arm()` call
  and is **not** preserved across deactivation or power cycles â€” every
  activation starts a fresh window.

No behaviour change on any path where `millis64()` does not advance between the
two former call sites (i.e., all normal execution).

### Stale test timing comments in `test_ams_pulse_safety.cpp`

**File:** `test/test_ams_integration/test_ams_pulse_safety.cpp`

Two inline comments in `test_pulse_safe_delay_blocks_early_fire` and
`test_pulse_safe_delay_allows_fire_after_delay` described timing semantics
from before `armAndTickAt` was updated to call `clock::advanceMs(1)` before
`arm()`:

| Before | After |
|--------|-------|
| `// arm() â†’ activationMs_ = ~0 ms. Tick immediately (1 ms elapsed).` | `// armAndTickAt: arm() at t=1 ms (activationMs_=1); elapsed at tick = 1 ms < 5000 ms safe_delay.` |
| `// arm() at t=0, tick at t=6000 ms (> 5000 ms safe_delay).` | `// armAndTickAt: arm() at t=1 ms (activationMs_=1); elapsed at tick = 6000 ms > 5000 ms safe_delay.` |

Both comments now match `armAndTickAt`'s documented contract: elapsed time
seen by the safe_delay gate equals exactly `tickMs` (not `tickMs + 1`).
No test logic changed.
