# ARES v2.3.9 — Changelog

**Released:** 2026-05-30
**Previous version:** 2.3.8
**Type:** Patch — safety rollback for pulse timer init; clang-tidy function-size warning resolved; code-review fixes (P0-4).

---

## Summary

This release closes the last outstanding P0 finding from the post-v2.3.7 audit
and eliminates the only remaining clang-tidy warning in the codebase.

**P0-4** — `PulseDriver::begin()` iterated over channels creating FreeRTOS
one-shot timers via `xTimerCreateStatic`.  If creation failed on channel N > 0,
the function returned `false` immediately, leaving timers `0..N-1` registered
in the FreeRTOS timer service but the driver in a state where `ready_` was
never set to `true`.  On a subsequent `begin()` call (unlikely in normal
operation but possible after a factory-reset or test-harness restart) those
orphaned handles would be leaked into the timer daemon's internal lists.

The fix adds an explicit rollback loop: on failure at channel `ch`, each timer
`i < ch` that was successfully created is removed from the timer service via
`xTimerDelete(timers_[i], pdMS_TO_TICKS(100U))`, then its slot is cleared to
`nullptr`.  After the function returns `false` the driver is in a fully defined
invalid state: `ready_ == false` and all `timers_[i] == nullptr`.  `xTimerDelete`
is appropriate for statically-allocated timers — it removes the timer from the
daemon's active/inactive lists without releasing memory (the `StaticTimer_t`
buffers are members of the driver object and remain valid for a future `begin()`
call).

**clang-tidy `readability-function-size`** — `setup()` in `src/main.cpp` was
81 lines (threshold: 80), triggering one clang-tidy warning that kept the CI
pipeline at WARN instead of a fully clean PASS.  The radio MAC key wiring block
(9 lines) was extracted into a new file-scope static helper
`applyRadioMacKey(DeviceConfig&, ares::RadioDispatcher&)`, following the same
pattern as the existing `applyBootCheckpoint` helper.  `setup()` is now 74
lines and the clang-tidy check passes cleanly.  No functional change.

## Detailed description

### P0-4: `PulseDriver::begin()` timer rollback (`pulse_driver.cpp`)

**Root cause.**  The creation loop for FreeRTOS one-shot timers had a missing
cleanup path:

```cpp
// Before
if (timers_[ch] == nullptr)
{
    LOG_E(TAG, "timer creation failed for channel %u", ...);
    return false;   // ← timers 0..ch-1 leaked into timer service
}
```

**Fix.**

```cpp
// After
if (timers_[ch] == nullptr)
{
    LOG_E(TAG, "timer creation failed for channel %u", ...);
    for (uint8_t i = 0U; i < ch; i++)
    {
        if (timers_[i] != nullptr)
        {
            (void)xTimerDelete(timers_[i], pdMS_TO_TICKS(100U));
            timers_[i] = nullptr;
        }
    }
    return false;
}
```

`xTimerDelete` sends a command to the timer daemon's command queue.  The
100 ms timeout is sufficient for the daemon to process the delete before
`begin()` returns.  Since `xTimerCreateStatic` uses caller-supplied
`StaticTimer_t` memory, `xTimerDelete` never attempts to `pvPortFree` the
buffer — the static lifetime of `timerBufs_[i]` is preserved.

**Postconditions after a failed `begin()`:**
- `ready_ == false` — `fire()` rejects all calls.
- `timers_[i] == nullptr` for all `i` — no orphaned handles.

### clang-tidy `readability-function-size`: `applyRadioMacKey` helper (`main.cpp`)

`setup()` contained an inline scoped block that loaded the provisioned HMAC-SHA256
key from `DeviceConfig` and installed it into `RadioDispatcher`.  At 81 lines the
function was one line over the configured threshold, triggering a clang-tidy warning
in every CI run.

The block was lifted into a named static helper with the same signature pattern as
`applyBootCheckpoint`:

```cpp
static void applyRadioMacKey(DeviceConfig& cfg, ares::RadioDispatcher& dispatcher)
{
    uint8_t radioKeyBuf[ares::proto::HMAC_KEY_LEN] = {};
    if (cfg.radioKey(radioKeyBuf, static_cast<uint8_t>(sizeof(radioKeyBuf))))
    {
        dispatcher.setMacKey(radioKeyBuf, static_cast<uint8_t>(sizeof(radioKeyBuf)));
    }
}
```

`setup()` now calls `applyRadioMacKey(deviceConfig, radioDispatcher)` — one line
replacing nine.  The stack-local `radioKeyBuf` is zeroed in the same stack frame
as before; no security properties changed.
