# ARES v2.2.0 — Changelog

**Released:** 2026-05-16  
**Previous version:** 2.1.2  
**Type:** Minor release — new AMS language feature; backward-compatible.

---

## Summary

v2.2.0 adds arithmetic expression support to the AMS `set` statement (AMS-4.8.8),
removing a key limitation that forced mission authors to consume transition slots
for derived-value calculations.  Prior to this release, statements such as:

```ams
set alt_agl  = BARO.alt - ground_alt
set vvel_avg = (BARO.alt - prev_alt) / 0.5
```

were rejected by the parser.  All existing `.ams` scripts remain fully compatible.

---

## Added

### AMS-4.8.8 — Arithmetic expression in `set` statements

The AMS `set` statement now accepts a right-hand side of the form:

```
set VARNAME = TERM op TERM [op TERM]
```

where:
- **TERM** is one of: `ALIAS.field` (sensor reading), a declared variable name,
  or a float literal.
- **op** is one of `+`, `-`, `*`, `/`.
- Two or three terms are supported (one or two operators).
- Parentheses are cosmetic and are stripped before tokenisation; they do not
  affect evaluation order.

**Evaluation order:** Strictly left-to-right.  `A - B / C` evaluates as
`(A - B) / C`.

**Parse-time rejections:**
- Literal zero in divisor position (`/ 0.0`) → parse error, `activate()` fails.
- Unknown peripheral alias in sensor term → parse error, `activate()` fails.

**Runtime error policy:**
- Invalid variable operand (`valid == false`), sensor read failure, runtime ÷ 0,
  or NaN/Inf result → `EVENT.warning(SENSOR_FAILURE)` emitted; target variable
  left unchanged.

**Scope:** Expressions are valid in `on_enter:`, `on_exit:`, and task `if:` set
actions.
