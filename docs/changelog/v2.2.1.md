`mission_script_engine_telemetry.cpp`.

**`[env:coverage]`** updated: HAL include paths, `ares_log.cpp` and
`ares_radio_protocol.cpp` added to build scope, `test_ams_helpers` added to
test filter.

Validated result: EXIT 0, 16 modules, line 78.0%, branch 67.6%, 371/371 tests PASSED.

## Improvements

### `run_coverage.ps1` — complete rewrite

**File:** `.vscode/scripts/run_coverage.ps1`

The previous script reported only line coverage per module with a TOTAL row.
The rewritten script adds four new capabilities:

#### Branch coverage (`gcov -b`)

`gcov` is now invoked with the `-b` flag, which instructs it to emit
`branch N taken X [fallthrough]` lines immediately after each conditional
source line in the `.gcov` output.  No compiler flag changes are required —
`-fprofile-arcs` was already present in the `[env:coverage]` build flags.

The summary table now includes two new columns per module:

```
File                                                 Cov/Exec    Line%    BrCov/Tot      Br%
---------------------------------------------------------------------------------------------------------
ares_radio_protocol.cpp.gcov                          147/150    98.0%        77/84    91.7%
mission_script_engine.cpp.gcov                        203/248    81.9%      163/208    78.4%
radio_dispatcher.cpp.gcov                             431/611    70.5%      207/279    74.2%
...
TOTAL                                                  3718      4768    78.0%     2722/4026    67.6%
```

Only line coverage is used for the per-module pass/fail threshold (`≥70%`).

#### Uncovered line numbers detail

A new `=== Uncovered line detail ===` section is appended after the summary
table.  For every module with at least one uncovered executable line, the
exact line numbers are listed (first 25; if more exist, a `(N total, first 25
shown)` note is added):

```
=== Uncovered line detail ===
  [ares_log]  4 uncovered line(s):
    40, 52, 66, 74
  [radio_dispatcher]  180 uncovered line(s):
    231, 232, 237, ... (180 total, first 25 shown)
```

#### GCOV exclusion markers

The parser now honours:

| Marker              | Scope                                         |
|---------------------|-----------------------------------------------|
| `GCOV_EXCL_LINE`    | Excludes the source line and its branch lines |
| `GCOV_EXCL_START`   | Starts an excluded block                      |
| `GCOV_EXCL_STOP`    | Ends an excluded block                        |

Branch lines generated by `gcov -b` (which follow the conditional source line)
inherit the exclusion state of their preceding source line via the
`$prevExclL` flag.

#### Coverage trend comparison

A new `=== Coverage trend ===` section compares the current run against the
previous `evidence/latest/coverage.txt` baseline:

```
=== Coverage trend ===
  Line coverage:   77.8% -> 78.0%  (+0.2%)
  Branch coverage: 67.1% -> 67.6%  (+0.5%)
```

On the first run after upgrading (i.e. when no previous branch data is stored),
branch coverage shows `(first run with branch data)  67.6%`.

#### TOTAL line format (backward-compatible)

The TOTAL line is extended with branch totals as a suffix, preserving the
format expected by `Parse-CoverageOutput` in the master pipeline:

```
TOTAL  3718  4768  78.0%   2722/4026  67.6%
```

The master pipeline regex `(\d+)\s+(\d+)\s+([\d.,]+)%` continues to extract
line coverage from the first three fields unchanged.

### `run_master_pipeline.ps1` — branch coverage in CI output

**File:** `.vscode/scripts/run_master_pipeline.ps1`

#### `Parse-CoverageOutput` extended

The function now returns three additional fields extracted from the TOTAL line:

| Field          | Type     | Description                       |
|----------------|----------|-----------------------------------|
| `BranchCov`    | `int`    | Covered branches                  |
| `BranchTotal`  | `int`    | Total executable branches         |
| `BranchPct`    | `double` | Branch coverage percentage        |

Branch data is extracted by matching the trailing `(\d+)/(\d+)\s+([\d.,]+)%\s*$`
pattern on the TOTAL line.  When absent (old-format report), the fields default
to `0` / `0` / `0.0`.

#### Coverage step detail string

The Coverage (gcov) CI step detail now reads:

```
3718/4768 lines  78.0%  |  2722/4026 branches  67.6% [SITL subset]
```

When no branch data is available (old report format), the detail falls back to
the previous format: `3718/4768 lines 78.0% [SITL subset]`.

### Test suite expansion

Four new test files have been added to increase coverage of previously
under-tested paths.  All new tests pass within the existing SITL/native
build pipeline (`pio test -e native`).

#### `test/test_ams_helpers/test_log_api.cpp` — 3 tests

Covers the three lines of `ares_log.cpp` not reached by any other suite:

| Test | Description |
|------|-------------|
| `test_setlevel_changes_runtime_level` | `setLevel()` persists and `getLevel()` reflects it |
| `test_emit_below_threshold_is_silent`  | `emit()` below current level produces no output    |
| `test_emit_at_threshold_is_visible`    | `emit()` at or above current level produces output |

#### `test/test_ams_integration/test_ams_coverage_ext.cpp` — 15 tests

SITL integration tests targeting gaps in three modules:

- **`mission_script_engine_parser_actions.cpp`** — `on_exit:` EVENT/CALIBRATE
  actions; `max`/`min` and `delta` set variants; unknown verb rejection.
- **`mission_script_engine_actions.cpp`** — runtime execution of CALIBRATE,
  delta-set, max/min-set; alarm monitoring trigger.
- **`mission_script_engine_sensor.cpp`** — GPS `lat`/`lon`/`speed`/`pressure`
  fields; IMU gyro fields in HK telemetry; bad-field parse rejection.

The test runner (`test_ams_integration/main.cpp`) now registers 157 tests total.

#### `test/test_dispatcher/test_command_dispatch.cpp` — 40 tests

Unit tests for `RadioDispatcher` non-fragmented COMMAND frame paths not
exercised by `test_fragment_reassembly`:

`REQUEST_STATUS`, `REQUEST_CONFIG`, `VERIFY_CONFIG`, `SET_MODE` (enable /
disable / invalid), `SET_FCS_ACTIVE`, `SET_CONFIG_PARAM`,
`REQUEST_TELEMETRY`, `SET_TELEM_INTERVAL`, `ARM_FLIGHT`, `ABORT`,
`FIRE_PULSE_A`, `FACTORY_RESET`, and `FLAG_ACK_REQ` ACK/NACK completion.

The test runner (`test_dispatcher/main.cpp`) now registers 45 tests total.

#### `test/test_ams_parser/test_ams_parser_cond_and_meta.cpp` — 9 tests

Parser corpus tests for condition types and metadata directives not reached
by the SITL integration suite:

| Test | Exercises |
|------|-----------|
| Falling/rising edge conditions | `parseConditionLocked` — falling/rising variants |
| Delta conditions               | `parseConditionLocked` — `delta` variant         |
| `radio.config` directive       | `parseMetaSectionLocked` — radio config parsing  |

### GCOV exclusion markers — `mission_script_engine_telemetry.cpp`

**File:** `src/ams/mission_script_engine_telemetry.cpp`

Three legacy code paths that are structurally unreachable under the
slot-based parser (added in v2.0.0) have been annotated with
`GCOV_EXCL_START` / `GCOV_EXCL_STOP` comments:

1. Legacy single-slot LOG path (`logSlotCount == 0` branch)
2. Dead legacy `appendLogReportLocked` call path

> **Note:** The analogous single-slot HK path (`sendHkReportLocked`) was
> initially excluded but the markers were removed after review — the function
> is still called as an explicit legacy fallback in `mission_script_engine.cpp`
> and `mission_script_engine_runtime.cpp` and its coverage should be tracked.

These annotations prevent the coverage report from counting those branches
as uncovered, giving a more accurate reflection of reachable code coverage.
