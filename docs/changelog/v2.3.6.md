# ARES v2.3.6 — Changelog

**Released:** 2026-05-27
**Previous version:** 2.3.5
**Type:** Patch — robustness: LED BOOT state guaranteed to exit on every setup() path (BUG-18).

---

## Summary

This release closes a latent diagnostic failure identified in the post-v2.3.5 audit.

**BUG-18** — `StatusLed` could remain stuck in the `BOOT` blink pattern indefinitely
if `setup()` returned before reaching `applyBootCheckpoint()` (e.g. a future
critical sensor-initialisation failure adding an early `return`).  The `BOOT`
pattern is visually identical across all fault conditions, giving the field
operator no indication of *why* the system is hung.

The fix has two parts:

1. **`StatusLed::getMode()`** — new thread-safe accessor that exposes the current
   LED mode from the same atomic used by `setMode()`.  Invalid raw values are
   normalised to `ERROR`, matching the internal `decodeOperatingMode()` logic.
   The class-level concurrency comment is updated to document `getMode()` as
   lock-free alongside `setMode()`.

2. **RAII `LedBootGuard` in `src/main.cpp`** — a file-scope struct (not a local
   class) whose destructor fires on **every** exit path from `setup()` (normal
   completion *and* any future early return).  If the LED mode is still `BOOT`
   when the destructor runs, it is forced to `ERROR`.  Declaring the guard at
   file scope also satisfies `cppcoreguidelines-special-member-functions`: all
   four copy/move specials are explicitly `= delete`d (CERT-18.3), and the
   destructor is the only user-defined special member.  This replaces the
   earlier post-`applyBootCheckpoint` `if`-check, which could not catch paths
   that skipped `applyBootCheckpoint()` entirely.

A secondary fix corrects an incorrect comment on the `statusLed.begin()` call
that described the initial LED state as "solid green (IDLE)" when it is in fact
"fast green blink (BOOT)".

## Changes

### `src/sys/led/status_led.h`

| Location | Change |
|----------|--------|
| Public API — new method | `ares::OperatingMode getMode() const` — thread-safe read of the current mode; normalises out-of-range values to `ERROR` |

### `src/sys/led/status_led.cpp`

| Location | Change |
|----------|--------|
| After `setMode()` | Implementation of `getMode()`: reads `mode_` atomic, applies range check against `OperatingMode::LAST`, returns `ERROR` for invalid values |

### `src/main.cpp`

| Location | Change |
|----------|--------|
| `statusLed.begin()` comment | Corrected from "solid green (IDLE)" to "fast green blink (BOOT)" |
| `setup()` — after `applyBootCheckpoint()` | Added safety-net: `if (statusLed.getMode() == BOOT) { statusLed.setMode(ERROR); }` — ensures any premature-return path leaves the LED in a diagnostically useful state |
