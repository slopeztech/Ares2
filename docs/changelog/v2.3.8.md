# ARES v2.3.8 — Changelog

**Released:** 2026-05-29
**Previous version:** 2.3.7
**Type:** Patch — security hardening: release-invisible safety checks replaced (P0-1, P0-5, P1-1); test-pipeline correctness fix.

---

## Summary

This release closes three latent safety issues identified in the post-v2.3.7 audit,
all sharing the same root cause: `ARES_ASSERT` used as the sole enforcement mechanism
for invariants that must also hold in release builds (`-DARES_NDEBUG`).

**P0-1** — `xSemaphoreCreateMutexStatic()` was validated exclusively via
`ARES_ASSERT`, which is a no-op in release builds (`-DARES_NDEBUG`).  On the
rare but possible case of a failed static mutex creation (corrupted TCB region,
stack exhaustion, or future refactor changing allocation), the firmware would
continue with a `nullptr` mutex.  Every subsequent `xSemaphoreTake(nullptr, ...)`
call either silently bypasses the critical section or crashes unpredictably
depending on FreeRTOS internals — with no visible indication to the operator.

The fix replaces the `ARES_ASSERT` with an explicit null check that logs a
`LOG_E` message and returns `false` from `begin()`, stopping the subsystem
cleanly on both debug and release builds.  `setup()` now treats these
failures as fatal: if `storageIf.begin()`, `missionEngine.begin()`, or
`apiServer.begin()` returns `false`, setup returns early and `LedBootGuard`
forces the `ERROR` operating mode.

Affected subsystems:
- **API server** (`cfgMtx_`) — guards all `PUT /api/device/config` and
  `GET /api/config` handler accesses to `DeviceConfig`.
- **LittleFS storage** (`mutex_`) — guards all filesystem read/write operations
  including mission scripts, logs, and device config persistence.

**P0-5** — `MissionUploadBuf::get()` enforced task ownership exclusively via
`ARES_ASSERT(xTaskGetCurrentTaskHandle() == expected)`, which is a no-op in
release builds.  Any future refactor routing the mission-upload handler through
a second task would introduce a silent data race on the 4 KB static buffer with
no runtime detection in production firmware.

The root fix is a new macro `ARES_REQUIRE(cond)` in `src/ares_assert.h` that
is structurally identical to `ARES_ASSERT` but is **never elided** — it aborts
the firmware in both debug and release builds.  `MissionUploadBuf::get()` now
uses `ARES_REQUIRE` for its task ownership check, making a violation impossible
to go undetected regardless of build configuration.

**P1-1** — `MissionScriptEngine::begin()` used `ARES_ASSERT(sBeginCount == 0U)` as
its only guard against a second call.  `ARES_ASSERT` is a no-op in release builds,
so in production firmware a double-begin (or a second engine instance) went
completely undetected.  The invariant matters because `tryRestoreResumePointLocked()`
uses a function-local static buffer (`static char buf[512]`) in `.bss` shared across
all invocations — two engines calling `begin()` concurrently would race on it.
Replaced with `ARES_REQUIRE`; the adjacent `ARES_ASSERT(mutex_ != nullptr)` (already
paired with a real null check below it) was removed as redundant.

**Test pipeline (CI)** — Introducing `ARES_REQUIRE` in P1-1 exposed a secondary bug:
`sBeginCount` was a function-local static that was never decremented, so every
test fixture that created a fresh `MissionScriptEngine` and called `begin()` aborted
on the second invocation (exit code 3 on Windows = `abort()`).  This caused three
test suites (`test_ams_integration`, `test_ams_parser`, `test_dispatcher`) to ERRORED
in the `sim` and `coverage` environments — both of which define `-DARES_NDEBUG`,
explaining why the original `ARES_ASSERT` never triggered.

The fix promotes `sBeginCount` to a file-scope static, adds a `bool begun_` member,
and adds `~MissionScriptEngine()` to decrement the counter when an engine that called
`begin()` is destroyed.  The invariant is unchanged: at most one engine may have
`begin()` outstanding at a time; sequential reuse is now permitted.

The master pipeline parser (`Parse-UnityOutput`) was also updated to understand
PlatformIO's aggregate output format (`N test cases: N succeeded`) in addition to
the classic Unity `X Tests Y Failures Z Ignored` line, so the CI detail column
shows actual test counts instead of "exit=0 (no summary line found)".

## Detailed description

### P0-5: `ARES_REQUIRE` macro + `MissionUploadBuf::get()` (`ares_assert.h`, `api_server.cpp`)

**Before (`ares_assert.h`):** no `ARES_REQUIRE` macro existed.

**After (`ares_assert.h`):**

```cpp
#define ARES_REQUIRE(cond)                                              \
    do {                                                                \
        if (!(cond)) {                                                  \
            LOG_E("REQUIRE", "%s:%d %s", __FILE__, __LINE__, #cond);   \
            Serial.flush();                                             \
            ::abort();                                                  \
        }                                                               \
    } while (0)
```

**Before (`api_server.cpp` — `MissionUploadBuf::get()`):**

```cpp
static char* get(TaskHandle_t expected)
{
    ARES_ASSERT(xTaskGetCurrentTaskHandle() == expected);  // no-op in release
    return buf_;
}
```

**After (`api_server.cpp` — `MissionUploadBuf::get()`):**

```cpp
static char* get(TaskHandle_t expected)
{
    ARES_REQUIRE(xTaskGetCurrentTaskHandle() == expected);  // active in release
    return buf_;
}
```

### P0-1: Static mutex validation (`api_server.cpp`, `littlefs_storage.cpp`)

**Before (`api_server.cpp`):**

```cpp
cfgMtx_ = xSemaphoreCreateMutexStatic(&cfgMtxBuf_);
ARES_ASSERT(cfgMtx_ != nullptr);  // no-op in release
```

**After (`api_server.cpp`):**

```cpp
cfgMtx_ = xSemaphoreCreateMutexStatic(&cfgMtxBuf_);
if (cfgMtx_ == nullptr)
{
    LOG_E(TAG, "mutex create failed");
    return false;
}
```

**Before (`littlefs_storage.cpp`):**

```cpp
mutex_ = xSemaphoreCreateMutexStatic(&mutexBuf_);
ARES_ASSERT(mutex_ != nullptr);  // no-op in release
```

**After (`littlefs_storage.cpp`):**

```cpp
mutex_ = xSemaphoreCreateMutexStatic(&mutexBuf_);
if (mutex_ == nullptr)
{
    LOG_E(TAG, "mutex create failed");
    return false;
}
```

### P1-1: AMS engine double-begin guard (`mission_script_engine.cpp`)

**Before:**

```cpp
static uint8_t sBeginCount = 0U;
ARES_ASSERT(sBeginCount == 0U);  // no-op in release
sBeginCount++;

mutex_ = xSemaphoreCreateMutexStatic(&mutexBuf_);
ARES_ASSERT(mutex_ != nullptr);  // redundant
if (mutex_ == nullptr)
{
    return false;
}
```

**After:**

```cpp
static uint8_t sBeginCount = 0U;
ARES_REQUIRE(sBeginCount == 0U);  // active in release
sBeginCount++;

mutex_ = xSemaphoreCreateMutexStatic(&mutexBuf_);
if (mutex_ == nullptr)
{
    return false;
}
```

### Test pipeline: `MissionScriptEngine` destructor + `sBeginCount` lifetime (`mission_script_engine.cpp`, `mission_script_engine.h`)

**Before:**

```cpp
// function-local static — never decremented
bool MissionScriptEngine::begin()
{
    static uint8_t sBeginCount = 0U;
    ARES_REQUIRE(sBeginCount == 0U);
    sBeginCount++;
    // ...
}
// No destructor declared.
```

**After:**

```cpp
// file-scope — destructor can decrement it
static uint8_t sBeginCount = 0U;

MissionScriptEngine::~MissionScriptEngine()
{
    if (begun_)
    {
        sBeginCount--;
    }
}

bool MissionScriptEngine::begin()
{
    ARES_REQUIRE(sBeginCount == 0U);
    sBeginCount++;
    begun_ = true;
    // ...
}
```

### Test pipeline: `Parse-UnityOutput` PlatformIO format (`run_master_pipeline.ps1`)

**Before:** only matched `X Tests Y Failures Z Ignored` (classic Unity format, not emitted by current PlatformIO versions).

**After:** also matches `N test cases: N succeeded` (PlatformIO aggregate line), so the CI report detail column shows real counts.

### `sBeginCount` rollback on init failure (`mission_script_engine.cpp`)

**Before:** `sBeginCount` incremented and `begun_` set to `true` before mutex creation.  On any failure path (`mutex_ == nullptr` or `guard.acquired() == false`), the counter was left incremented and the destructor would not decrement it, permanently blocking a retry.

**After:**

```cpp
ARES_REQUIRE(sBeginCount == 0U);
sBeginCount++;

mutex_ = xSemaphoreCreateMutexStatic(&mutexBuf_);
if (mutex_ == nullptr)
{
    sBeginCount--;
    return false;
}

ScopedLock guard(mutex_, pdMS_TO_TICKS(ares::AMS_MUTEX_TIMEOUT_MS));
if (!guard.acquired())
{
    sBeginCount--;
    return false;
}

(void)tryRestoreResumePointLocked(millis64());
begun_ = true;   // set only on success
return true;
```

---

### `begin()` return values handled in `setup()` (`main.cpp`)

**Before:** all three critical `begin()` calls discarded their return value with `(void)`.

**After:**

```cpp
if (!storageIf.begin())
{
    LOG_E("BOOT", "storage init failed");
    return;
}
// ...
if (!missionEngine.begin())
{
    LOG_E("BOOT", "AMS engine init failed");
    return;
}

if (!apiServer.begin())
{
    LOG_E("BOOT", "API server init failed");
    return;
}
```

Early return triggers `LedBootGuard::~LedBootGuard()`, which forces `OperatingMode::ERROR` on the status LED.
