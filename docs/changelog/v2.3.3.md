# ARES v2.3.3 — Changelog

**Released:** 2026-05-22
**Previous version:** 2.3.2
**Type:** Patch — security: radio link HMAC-SHA256 authentication ([C1]).

---

## Summary

This release implements the last open critical security finding from the radio
protocol audit: **[C1] — unauthenticated command injection over the LoRa link**.

Prior to this release any node within radio range could inject COMMAND frames
(FC = 0x01) that would be dispatched directly to the AMS engine, flight
handlers, and device-config endpoints with no authentication.

The fix adds truncated HMAC-SHA256 (8 bytes, RFC 2104) as a mandatory MAC tag
on every inbound COMMAND frame.  A 16-byte pre-shared key is provisioned at
first boot alongside the WiFi password and API token (generated by
`provisionIfFirstBoot()`, introduced in v2.3.2).  The key is stored in
`/ares_device.json` as a 32-character lowercase hex string and is never
exposed through the REST API.

Key design decisions:
- **No external crypto library** — a self-contained portable SHA-256
  (FIPS 180-4) + HMAC (RFC 2104) is implemented in `radio_mac.cpp` (~200 LOC).
  This avoids pulling in mbedTLS / wolfSSL for a single primitive, is
  fully auditable, and compiles cleanly on both ESP32 (Arduino) and the
  native test environment.
- **8-byte truncated tag** — the 64-byte APUS payload budget is finite.
  8 bytes (64 bits) meets the NIST SP 800-107 guidance for truncated MACs
  and adds only 8 bytes to COMMAND frames.
- **FLAG_MAC (bit 4 of FLAGS)** — the new flag signals that the final 8 bytes
  of the payload carry the MAC tag.  The tag is stripped from the frame before
  it is forwarded to handlers.  Non-COMMAND frame types are unaffected.
- **Open mode** — if no key has been provisioned (`radioKeyHex_` is empty)
  `RadioDispatcher` forwards COMMAND frames without MAC checking, preserving
  backward-compatibility during development and bench testing.
- **13 new unit tests** covering RFC 4231 test vectors, key/message
  distinguishability, constant-time reject, `appendCommandMac`,
  and `verifyCommandMac` round-trip / wrong-key / no-flag cases.

## Changes

### `src/comms/radio_mac.h` *(new file)*

| Declaration | Purpose |
|-------------|---------|
| `namespace ares::proto` | |
| `constexpr uint8_t HMAC_LEN = 8U` | Wire size of the truncated MAC tag |
| `constexpr uint8_t HMAC_KEY_LEN = 16U` | Pre-shared key length in bytes |
| `computeHmac8(key, keyLen, msg, msgLen, mac)` | Computes 8-byte truncated HMAC-SHA256 |
| `verifyHmac8(key, keyLen, msg, msgLen, expected)` | Constant-time verification |
| `appendCommandMac(key, keyLen, Frame&)` | Sets `FLAG_MAC`, appends tag to COMMAND frame |
| `verifyCommandMac(key, keyLen, const Frame&)` | Verifies tag; returns false on any failure |

### `src/comms/radio_mac.cpp` *(new file)*

- Self-contained SHA-256 (FIPS 180-4) block-compression function.
- HMAC-SHA256 (RFC 2104) using two SHA-256 passes over ipad/opad.
- `buildMacInput()` serialises the MAC-covered fields:
  `[VER(1), FLAGS(1), NODE(1), TYPE(1), SEQ(1), cmd_len(1), payload[0..cmd_len-1]]`
  with `FLAG_MAC` asserted in the FLAGS byte before hashing.
- No heap allocation (PO10-3 compliant); all buffers are stack-local.
