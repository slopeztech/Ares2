# ARES v2.3.10 ŌĆö Changelog

**Released:** 2026-05-31
**Previous version:** 2.3.9
**Type:** Patch ŌĆö API server hardening (P0-2, P0-3, P1-5, P1-6); HTTP parser fuzz tests (P3-4); MISRA/clang-tidy CI clean.

---

## Summary

This release closes five code-review findings from the post-v2.3.9 audit and
brings the CI pipeline to a fully clean PASS (no warnings).

**P0-2** ŌĆö `ApiServer::begin()` called `xTaskCreateStaticPinnedToCore()` and
relied on `ARES_ASSERT` (elided in release builds) to catch a null task handle.
In a release binary a failed task creation would silently continue, leaving the
server in an uninitialised state.  The assert is replaced by an explicit runtime
check that logs an error, signals the status LED to `ERROR` mode, and returns
`false` to the caller.

**P0-3** ŌĆö `s_corsHeadersBuf` was written inside `refreshCorsHeaders()` without
holding `cfgMtx_`.  Any concurrent call to `GET /api/status` or a config-reload
could observe a partially-written CORS header string.  The fix conditionally
acquires `cfgMtx_` via `ScopedLock`: the first call from `begin()` occurs before
the mutex is created (single-threaded setup phase) and writes directly; all
subsequent calls from the running task acquire the lock first.  No raw FreeRTOS
macro calls appear in `api_server.cpp` ŌĆö the `dangerousTypeCast` suppression
lives only in `rtos_guard.h` as before.

**P1-5** ŌĆö `uxTaskGetStackHighWaterMark(nullptr)` queries the *calling task's*
stack, not the API task's stack.  During the brief window between `begin()`
returning and the API task being scheduled, `nullptr` resolves to `loop()`'s
stack, producing a spurious low-watermark log line.  Changed to
`uxTaskGetStackHighWaterMark(taskHandle_)`.

**P1-6** ŌĆö `esp_task_wdt_reset()` was called at the *end* of the `while(true)`
loop body.  If `httpServer.accept()` or any subsequent call blocked longer than
the WDT timeout (5 s, project default), the watchdog would fire before the reset
was reached.  Moving the reset to the *top* of the loop feeds the WDT
unconditionally on every iteration, independent of what happens inside the body.

**P3-4** ŌĆö 10 new HTTP parser robustness tests in `test/test_api_routing/` inject
adversarial byte streams directly into `ApiServer::handleClient()` (promoted to
`public` for test access) and verify correct status codes without a live TCP
stack or RTOS task: empty request, bare CRLF, LF-only line ending, negative
`Content-Length`, oversized `Content-Length`, truncated body, body without
`Content-Type`, URI > 64 chars, 20 headers (capped at 16), and a header line
without a colon separator.

**MISRA Cppcheck** ŌĆö `xSemaphoreTake`/`xSemaphoreGive` macro internals expand a
C-style cast that Cppcheck flags as `dangerousTypeCast`.  All FreeRTOS macro
call sites in `api_server.cpp` were eliminated by delegating to `ScopedLock`,
which already carries the inline suppression in `rtos_guard.h`.

**clang-tidy `readability-function-size`** ŌĆö `PulseDriver::begin()` was 87 lines
with nesting depth 6 (thresholds: 80 lines / 5 levels).  The GPIO-init loop was
extracted into a new private helper `initGpioPins()` and the timer-rollback loop
into `deleteTimers(uint8_t upToExclusive)`.  `begin()` is now ~40 lines with
maximum nesting depth 3.  No functional change.

## Detailed description

### P0-2: `begin()` task-creation error handling (`api_server.cpp`)

**Root cause.**  The original code used `ARES_ASSERT` to guard the task handle:

```cpp
taskHandle_ = xTaskCreateStaticPinnedToCore(...);
ARES_ASSERT(taskHandle_ != nullptr);   // elided with -DARES_NDEBUG
```

`ARES_ASSERT` compiles away in release builds (`-DARES_NDEBUG`).  A failed
`xTaskCreateStaticPinnedToCore` returns `NULL` (documented: insufficient stack
or scheduler not started); without the assert the function would fall through
and return `true`, leaving the server in an uninitialised state with no task
running.

**Fix.**

```cpp
if (taskHandle_ == nullptr)
{
    LOG_E(TAG, "API task creation failed ŌĆö degrading to no-WiFi mode");
    if (statusLed_ != nullptr) { statusLed_->setMode(ares::OperatingMode::ERROR); }
    return false;
}
```

The check is unconditional and never elided.

### P0-3: `s_corsHeadersBuf` mutex protection (`api_server.cpp`)

**Root cause.**  `refreshCorsHeaders()` wrote to `s_corsHeadersBuf` (a
file-scope `char` array) without holding `cfgMtx_`.  The API task calls
`refreshCorsHeaders()` on every `PUT /api/config` while the main loop and
any concurrent HTTP client may simultaneously read the buffer.

**Constraint.**  `ScopedLock` calls `ARES_ASSERT(m != nullptr)` on construction,
so it cannot be used unconditionally ŌĆö the first call from `begin()` happens
before `cfgMtx_` is created.

**Fix.**

```cpp
void ApiServer::refreshCorsHeaders()
{
    if (cfgMtx_ == nullptr)
    {
        devCfg_.buildCorsHeader(s_corsHeadersBuf, sizeof(s_corsHeadersBuf));
        return;
    }
    ScopedLock lk(cfgMtx_, pdMS_TO_TICKS(ares::API_CFG_MUTEX_TIMEOUT_MS));
    devCfg_.buildCorsHeader(s_corsHeadersBuf, sizeof(s_corsHeadersBuf));
}
```

The `dangerousTypeCast` suppression for FreeRTOS macro internals remains
exclusively in `rtos_guard.h`; `api_server.cpp` no longer contains any direct
`xSemaphoreTake`/`xSemaphoreGive` calls.

### P1-5: Stack watermark uses correct task handle (`api_server.cpp`)

`uxTaskGetStackHighWaterMark(NULL)` is defined by FreeRTOS as "query the
calling task".  Called from `begin()` (which runs in `loop()`'s task context),
it returned the watermark of `loop()`, not of the API task.  After the API task
is created and scheduled, the correct call is:

```cpp
LOG_D(TAG, "stack HWM: %u words", uxTaskGetStackHighWaterMark(taskHandle_));
```

---

### P1-6: WDT reset at top of run-loop (`api_server.cpp`)

**Before** ŌĆö WDT fed only after all work in the iteration completed:

```cpp
while (true)
{
    WiFiClient client = httpServer.accept();
    // ... potentially blocking work ...
    esp_task_wdt_reset();   // ŌåÉ could be reached too late
    vTaskDelay(...);
}
```

**After** ŌĆö WDT fed unconditionally at the start of every iteration:

```cpp
while (true)
{
    esp_task_wdt_reset();   // ŌåÉ first statement; always reached
    WiFiClient client = httpServer.accept();
    // ...
    vTaskDelay(...);
}
```

### P3-4: HTTP parser fuzz layer (`test_api_routing/test_http_parser.cpp`)

Ten tests cover edge cases in `parseRequestLine`, `parseHeaders`, and
`readRequestBody` by injecting raw byte streams via `SimWiFiClient`:

| ID  | Input | Expected response |
|-----|-------|-------------------|
| HP1 | Empty request (zero bytes) | No response sent |
| HP2 | Bare `\r\n` only | No response sent |
| HP3 | LF-only line ending (`GET / HTTP/1.1\n`) | 200 OK |
| HP4 | `Content-Length: -1` | 413 |
| HP5 | `Content-Length: 999999999` | 413 |
| HP6 | `Content-Length: 5` with only 3 body bytes | 400 |
| HP7 | Body present, no `Content-Type` header | 400 |
| HP8 | URI of 65 characters | 414 |
| HP9 | 20 headers (parser cap 16) | 200 (extra headers silently dropped) |
| HP10 | Header line without `:` separator | 200 (malformed header ignored) |

`handleClient()` was promoted from `private` to `public` in `api_server.h`
solely to enable this injection pattern; it is not part of the external API.

### clang-tidy: `PulseDriver::begin()` decomposition (`pulse_driver.cpp`)

`begin()` was 87 lines (threshold 80) with nesting depth 6 (threshold 5) due
to the GPIO-init loop and the timer-rollback loop both being inline.

Two private helpers were extracted:

```cpp
void PulseDriver::initGpioPins();
void PulseDriver::deleteTimers(uint8_t upToExclusive);
```

`initGpioPins()` configures fire and continuity GPIO directions for all wired
channels.  `deleteTimers(upToExclusive)` removes FreeRTOS timers `[0,
upToExclusive)` from the timer service during rollback.  `begin()` calls both
and is now ~40 lines with maximum nesting depth 3.  Behaviour is identical.
